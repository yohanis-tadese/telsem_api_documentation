Developer Guidelines

  • Use valid IDs: Ensure all API requests that reference entities like system users, groups, or categories use valid IDs (e.g., 7-digit numbers) that are generated by the backend. These IDs are returned in the API responses and should not be hardcoded.
  • No ID in request bodies: Do not include the id field in POST request bodies. IDs are automatically generated by the backend during the creation of resources and should only be referenced in URL parameters for update, delete, or read operations.
  • Dependency on other entities: Ensure you retrieve valid dependencies (like group_id for system users or lookupCategory_id for lookups) from their respective APIs before creating or updating records.
  • Handle success and failure responses: Always check for success (status codes 200-201) and failure (e.g., 400, 404) responses. Handle these responses appropriately in the frontend with user-friendly error or success messages.
  • Token-based authentication: For endpoints that require authentication, include the valid JWT token in the Authorization header: Authorization: Bearer [token].
  • Data formatting: Ensure all request data is correctly formatted according to the API requirements (e.g., valid phone number formats, email addresses, etc.). Check API documentation for field validations.
